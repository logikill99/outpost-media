Large Language Models — How They Actually Work

Category: AI/ML · Reading time: ~12 min · Written for Outpost

You've been using these things for years now. Maybe you've built applications on top of them. But do you actually know what's happening under the hood when you prompt an LLM? This is the honest technical explainer — written for someone who can read code but hasn't necessarily studied ML.

Start Here: What a Neural Network Is

A neural network is a function that takes numbers in and produces numbers out. That's the whole thing. The "neural" part is a historical marketing term that gestures vaguely at neurons, but what's actually happening is: stacked matrix multiplications, with non-linear functions (activation functions) in between.

The simplest neural network, a single layer, does this:

output = activation(weights × input + bias)

Stack dozens or hundreds of these layers with learned weights, and you have a "deep" neural network. The weights are the parameters. A model with "70 billion parameters" has 70 billion floating point numbers that define its behavior. Training is the process of finding the right values for all those numbers.

How Training Works (the Essentials)

You show the network an input, it produces an output, you compare that output to what you wanted (using a loss function), and you use calculus (backpropagation) to figure out how to adjust each weight to make the output closer to correct. Do this billions of times on enormous datasets and the weights converge to something useful.

This sounds almost too simple to produce intelligent behavior. It is, in fact, almost too simple — and yet it works spectacularly, for reasons that are still not fully understood even by the researchers building these systems.

Transformers and the Attention Mechanism

LLMs use a specific architecture called the Transformer, introduced in the 2017 paper "Attention is All You Need" (Vaswani et al., Google Brain). Before Transformers, language models used recurrent networks (RNNs/LSTMs) that processed text one token at a time, left to right. Transformers process all tokens simultaneously and use a mechanism called attention to relate each token to every other token in the context.

Tokens, Not Words

LLMs don't read words or characters — they read tokens. A token is roughly 3–4 characters of typical English text, but this varies significantly. "Transformer" might be one token. "Antidisestablishmentarianism" might be several. Common words are usually one token; rare words get split up. The model's "vocabulary" is typically 32,000–100,000+ tokens.

Attention: The Key Insight

Attention is the mechanism that lets a model figure out which parts of the input are relevant to each other. When processing the word "it" in "The trophy didn't fit in the suitcase because it was too big," attention lets the model learn that "it" relates to "trophy" (not "suitcase").

Mathematically, attention computes a weighted sum over "values" (vectors representing each token), where the weights come from comparing "queries" (what am I looking for?) to "keys" (what does each token represent?). The formula looks like:

Attention(Q, K, V) = softmax(QKT / √d_k) × V

Multi-head attention runs this in parallel multiple times with different learned projections, allowing the model to attend to different "aspects" of relationships simultaneously. GPT-4 has 96 attention heads per layer. Each head can specialize in different syntactic or semantic relationships.

The Full Transformer Stack

A Transformer language model is essentially:

  1. Token embedding (map token IDs to high-dimensional vectors)
  2. Positional encoding (add information about where in the sequence each token is)
  3. N × transformer blocks, each containing:
    • Multi-head self-attention
    • Feed-forward network (two linear layers with nonlinearity)
    • Layer normalization
    • Residual connections
  4. Final linear layer + softmax → probability distribution over vocabulary

The model predicts the probability of each token in the vocabulary being the "next" token. That's it. The entire apparent intelligence of a system like GPT-4 emerges from predicting what token comes next, trained on enough text at enough scale.

Training on Text Data

The base training for modern LLMs uses a massive corpus of text from the internet, books, code repositories, scientific papers, and more. We're talking petabytes of text. Common Crawl (a web scrape updated monthly) is a primary source. Books corpora. GitHub. Wikipedia. Academic papers. The data is filtered, deduplicated, and quality-scored, but at the end of the day it's mostly "a large chunk of everything humans have written in digital form."

The training objective is simple: predict the next token. Given a trillion tokens of text as training data, and a large enough model, the model has to learn something about language, facts, reasoning, and structure to do this accurately. The representation of all that implicit knowledge lives in the weights.

Compute and Scale

The "bitter lesson" of ML, attributed to Richard Sutton, is that general methods that scale with compute beat clever specialized methods. This has proven true with LLMs. Scaling laws (Kaplan et al., 2020; Hoffmann et al., 2022 "Chinchilla") showed that model performance improves predictably as you increase model size, training data, and compute — following power laws.

GPT-3 (2020): 175B parameters, ~45TB of text, ~3,640 petaflop-days of compute. State-of-the-art models today are significantly larger and have consumed orders of magnitude more compute. Training a frontier LLM costs tens to hundreds of millions of dollars in cloud compute.

Emergent Capabilities

One of the genuinely strange phenomena in LLM scaling is emergence — capabilities that appear relatively suddenly at certain model scales, not gradually. A model at 10B parameters can't do multi-step arithmetic reliably. A model at 100B+ parameters can. The capability appeared, but it wasn't explicitly trained for.

Current understanding: emergent capabilities appear when the model has enough capacity to implicitly form internal representations sophisticated enough to support the capability. The word "emerges" is sometimes overloaded — these aren't magical capabilities appearing from nowhere, they're capabilities that require a certain level of representational complexity to be useful, and models below that threshold appear to not have them.

Other emergent capabilities observed at scale: chain-of-thought reasoning, code generation, translation between many language pairs, few-shot learning from examples in the prompt. These weren't directly trained for — they emerged from predicting text at scale.

RLHF — How Models Get "Good"

A base language model trained on internet text will produce outputs that statistically resemble that training data. This includes harmful content, incorrect information presented confidently, and generally outputs that aren't actually useful for human interaction. The solution is Reinforcement Learning from Human Feedback (RLHF).

The process (simplified):

  1. Supervised fine-tuning: Show the model examples of good responses, written by humans. Fine-tune on these.
  2. Reward model training: Have humans compare pairs of outputs and rate which is better. Train a "reward model" to predict these human preferences.
  3. RL optimization: Use the reward model as a signal to further fine-tune the language model via reinforcement learning (specifically PPO, typically). The model learns to produce outputs that the reward model rates highly.

RLHF is why ChatGPT or Claude seems helpful, harmless, and honest compared to a raw language model. It's also why models sometimes refuse requests — the RLHF training included negative rewards for certain types of outputs. The tradeoffs in RLHF are non-trivial: optimizing too hard against the reward model can produce models that are sycophantic, or that have lost capability in some areas.

More recent approaches: DPO (Direct Preference Optimization) removes the explicit reward model. Constitutional AI (Anthropic) adds rule-based self-critique in the loop. The basic idea of human feedback shaping model behavior is consistent across these variants.

How Inference Works

When you prompt a model, here's what actually happens:

  1. Your text is tokenized into a sequence of integer IDs
  2. These tokens are embedded into vectors
  3. The vectors pass through all N transformer blocks (forward pass)
  4. The output layer produces a probability distribution over the vocabulary
  5. A token is sampled from this distribution (or the highest-probability token is selected — this is "greedy" decoding)
  6. That token is appended to the input and the process repeats

Each forward pass costs roughly O(N) in compute where N is context length, because attention has to relate every token to every other token. This is why longer contexts are significantly more expensive. A 128K context window costs much more per token than a 4K context window.

Modern inference systems use KV-cache: the "key" and "value" matrices computed during attention for past tokens are cached and reused, so each new token only requires computing attention for the new position against cached past positions. This dramatically reduces the per-token cost for long generations.

Parameters are stored in VRAM. A 70B parameter model in float16 requires ~140GB of VRAM minimum. Quantization (reducing precision to int8 or int4) can halve or quarter this at some quality cost. Running LLaMA 3 70B on a MacBook Pro M4 Max (128GB unified memory) is feasible with 4-bit quantization — this is wild compared to 5 years ago.

Current Capabilities and Limitations

What LLMs Are Genuinely Good At

  • Code generation, review, and explanation — often exceeds junior developer ability
  • Synthesis and summarization of complex documents
  • Translation, including low-resource language pairs that pre-LLM systems were weak on
  • Structured information extraction from unstructured text
  • Conversational reasoning across many domains
  • Instruction following with complex, multi-step requirements

What LLMs Still Get Wrong

  • Hallucination: Producing confident, fluent, plausible-sounding text that is factually incorrect. This is a fundamental property of autoregressive generation, not a bug to be patched.
  • Arithmetic and formal reasoning: Getting better with tool use (code interpreter), but native arithmetic is unreliable for complex calculations
  • Long-range consistency: Very long documents still expose context degradation — the model "forgets" (in a sense) early content
  • Reasoning about novel situations: Pattern-matching covers enormous ground but novel genuine reasoning problems can fail in unexpected ways
  • Knowledge cutoff: Training data has a cutoff date. The model doesn't know what happened last week unless told via retrieval or context

The line between "impressive emergent reasoning" and "sophisticated pattern matching over training data" is genuinely contested, philosophically and empirically. The practical upshot: treat LLM outputs as a smart first draft, not as ground truth, particularly for factual claims about specific events or numerical calculations.

What we've built is extraordinary. What it actually is — whether it "understands" in any meaningful sense — remains one of the most interesting open questions in science.
