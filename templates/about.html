📖 ABOUT THIS THING

What you're connected to, why it exists, and who made it.

The short version

You're connected to a tiny computer someone brought to the race. That computer is running its own WiFi network, and everything you see here (movies, games, live chat, race info) is being served from it with no internet connection involved. It's entirely self-contained. No cell signal required.

Why this exists

Cell service at F1 events is, famously, terrible. Forty thousand people all trying to load the same apps at once does what you'd expect. The idea here was simple: instead of fighting the network, just bring your own. Load the content you'd actually want at a race onto a device small enough to fit in a pocket, put it in AP mode, and let anyone nearby connect to it.

It started as a weekend project and got a little out of hand. The race info pages, the CTF challenge, the offline library of articles and books, movies for downtime between sessions, the realtime chat room. Worth it.

Partly it was an experiment in how much you could actually run on basically nothing. The answer turns out to be: more than you'd think.

Who made it

A software engineer who's also an F1 fan. Built this as a personal project for the race. No company, no product, no pitch.

How it works (plain English)

Your phone or laptop connected to a WiFi network called something like "The Dark Side of T18." That network isn't a router with an internet connection. It's a Raspberry Pi Zero 2W, which is a $15 credit-card-sized computer, running software that makes it pretend to be a WiFi access point.

When you connected, your device probably showed a "sign in to network" prompt (that's the captive portal). Tapping that, or navigating to any URL, brought you here. There's no internet connection, but there doesn't need to be. Everything you can see on this server is stored directly on a 64GB SD card inside the Pi. Movies, articles, race data, the chat backend, all of it.

Power draw is around 2 watts. It runs off a USB battery bank. The whole thing fits in a pocket.

How it works (technical)

Hardware is a Raspberry Pi Zero 2W (BCM2710A1, quad-core ARM Cortex-A53 at 1GHz, 512MB RAM). Single wlan0 interface running in AP mode via hostapd. DHCP and wildcard DNS served by dnsmasq, which resolves all queries to 10.0.0.1 so the captive portal fires reliably across iOS, Android, and Windows. Caddy on port 80 handles the reverse proxy and static file serving; it catches OS-specific captive portal probe URLs (/hotspot-detect.html, /generate_204, etc.) and redirects them to the portal page before Flask ever sees the request.

The app itself is Flask with Flask-SocketIO and eventlet. SQLite via SQLAlchemy. Content data is a mix of pregenerated JSON (F1/F2/PCCNA standings and driver data) and pre-scraped Wikipedia HTML pages served offline. Frontend is Alpine.js with no build step. Video files are served directly by Caddy with range-request support for seek. SQLite has WAL mode enabled because the Pi Zero 2W's SD card write latency under concurrent SocketIO and CTF traffic is not great in journal mode.

The CTF is six flag challenges seeded into SQLite at first boot. Flags are stored as SHA-256 hashes. Leaderboard is aggregated on read. The chat is SocketIO rooms persisted to SQLite with a 50-message history window. The whole thing was built in roughly two weeks of evenings with Claude Sonnet doing a lot of the data pipeline and template work while I handled architecture, deployment, and debugging.

Stack summary: Python 3.13 / Flask / Flask-SocketIO / SQLAlchemy / SQLite / Caddy / Alpine.js / hostapd / dnsmasq / Raspberry Pi OS Lite.

Troubleshooting

The page isn't loading at all. Make sure your device is connected to the Outpost WiFi and not falling back to cellular or your home network. Some phones will silently prefer a known network with internet access. Disable cellular or toggle WiFi off and back on if you're not getting the captive portal prompt.

I'm connected but can't reach the page. Try navigating directly to http://10.0.0.1 in your browser. Some devices cache a "no internet, don't bother" state and won't load even if the server is there. Typing the IP directly usually bypasses that.

Videos are buffering or won't play. The Pi is streaming video over WiFi from an SD card, so there's a real throughput ceiling. If more than three or four people are streaming simultaneously it'll start to bog down. Give it a few seconds to buffer, or wait for lighter traffic. The short films load fast, the feature films need a moment.

The chat shows an error or won't connect. The chat uses WebSockets, which occasionally drop on mobile browsers if the device sleeps briefly. A page refresh reconnects it.

The page looks broken with no styling. This usually means your browser cached a failed request for the CSS. Hard refresh (Ctrl+Shift+R on desktop, or clear the browser cache) and it should come back.
